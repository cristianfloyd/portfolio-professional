# Definición de Arquitectura de Contenidos: Proyectos vs. Challenges

Este documento define la línea divisoria estratégica entre los "Proyectos/Estudios de Caso" y los "Coding Challenges"

## La Línea Divisoria

| Característica       | **Proyecto / Case Study**                              | **Challenge / Kata**                              |
| :------------------- | :----------------------------------------------------- | :------------------------------------------------ |
| **¿Para qué sirve?** | Demostrar que resuelves problemas complejos.           | Demostrar que escribes código excelente.          |
| **Foco principal**   | **Impacto**: Ahorro de tiempo, seguridad, negocio.     | **Calidad**: SOLID, Patrones, Testing.            |
| **Formato**          | Explicación detallada (Storytelling) + Métricas.       | Código directo, Readme técnico y Tests.           |
| **Audiencia**        | Líderes de proyecto / Managers / Clientes.             | Tech Leads / Desarrolladores Senior.              |
| **Origen**           | Trabajo profesional (UBA), Freelance, Infraestructura. | Pruebas técnicas, Katas, Ejercicios de comunidad. |

## Aplicación en el Portfolio

### 1. Proyectos (Case Studies)

- **Ubicación**: Sección "Impacto Técnico & Resoluciones" en la Home.
- **Narrativa**: "Tenía un problema X, apliqué mi experiencia senior Y, y obtuve un resultado Z. Easy".
- **Páginas**: `/case-studies/[slug]`.

### 2. Challenges (Coding Challenges)

- **Ubicación**: Sección `/challenges`.
- **Narrativa**: "Este es un ejemplo de cómo escribo código. Aplicación rigurosa de principios de ingeniería".
- **Sincronización**: Automatizada desde GitHub mediante script.
